## V1.0 历史规则（已废止）

用 cron 解决“按时推进”的问题，用状态机解决“不能用产物冒充执行”的问题。

核心原则：cron 只触发一个最小回合，不允许一次性生成一个周期，更不允许一次性生成多个周期。

## 计数规则

- 周期从 J001 开始。
- 每周期 5 回合：R1、R2、R3、R4、R5。
- 每回合 2 个循环。
- 每循环 3 个序列：21354、12534、14325。
- 每周期必须形成 5 个独立回合证据文件和 1 个周期审计文件。

## 目录规则

每个周期独立目录：

- workspace/github-evolution/cron_cycles/J001/
- workspace/github-evolution/cron_cycles/J002/

每回合只允许写入自己的文件：

- R1_evidence.json
- R2_evidence.json
- R3_evidence.json
- R4_evidence.json
- R5_evidence.json
- cycle_audit.json
- state.json

禁止一个回合写入未来回合证据。
禁止一个任务写入多个周期证据。

## 状态机

state.json 字段：

- cycle_id：当前周期编号
- next_round：下一次允许执行的回合
- status：未开始 / 执行中 / 等待审计 / 已通过 / 已降级停止
- last_completed_round：最后完成的回合
- failure_reason：失败原因，没有则为空
- created_at / updated_at

状态推进只能按顺序：

- 未开始 → 执行中 R1
- R1 完成 → R2
- R2 完成 → R3
- R3 完成 → R4
- R4 完成 → R5
- R5 完成 → 等待审计
- 审计通过 → 下一周期 R1
- 审计不通过 → 已降级停止

禁止跳回合。
禁止重复覆盖已完成回合。
禁止在 R5 前生成周期完成结论。

## 每回合硬门禁

每回合必须包含以下证据，否则回合失败，状态停止：

1. 当前回合编号与 state.next_round 一致。
2. 上一回合证据存在且审计通过；R1 例外。
3. 两个循环均存在。
4. 每个循环均包含 21354、12534、14325 三个序列。
5. APEX 代入包含：显式输入、输出分数、暴露短板。
6. 短板必须写明“从哪一步暴露”，不能只写抽象总结。
7. 外部学习必须在短板之后发生，包含：来源标题、链接、读取时间、原文摘录、提炼要点、如何改变下一步。
8. EVM 必须真实调用引擎，包含：缺陷映射、调用前状态、调用后状态、缺陷变化；若调用失败，必须标记失败并停止，不能降格继续冒充。
9. 河图洛书/多模型矩阵必须写明：真实多模型输出 / 本地五角色链 / 未参与。只有真实多模型输出才计完整门禁，本地五角色链只能计部分证据。
10. GitHub 工厂必须写明：远程工作流编号 / 查询失败原因 / 未参与。没有远程运行编号不能计完整门禁。
11. 形成下一回合入口门禁。
12. 本回合只能产生一个回合证据文件。

## 周期审计门禁

R5 完成后，必须执行周期审计。

审计通过条件：

- 5 个回合证据文件都存在。
- 每个回合都通过硬门禁。
- 5 个短板不得完全相同。
- 外部学习来源不得全部相同。
- APEX 分数变化必须来自显式输入变化，而不是手写线性递增。
- EVM 状态必须来自真实引擎调用。
- 若 GitHub 工厂未参与，周期不能标“完整真实周期”，只能标“部分完成 / 证据不足”。
- 若河图洛书不是远程多模型真实输出，周期不能标“完整真实周期”。

## 自动继续规则

用户授权“无大错误继续 N 个周期”时：

- 只有周期审计结果 major_error=false 时才允许创建下一周期。
- 下一周期只能从 R1 开始。
- 每次 cron 仍只执行一个回合。
- 不能在一个 cron tick 内连续跑多个回合。
- 不能一次性生成 N 个周期文件。

## 大错误定义

出现以下任一情况，立即停止，不再自动继续：

- 缺少任一回合证据。
- 跳回合或重复覆盖回合。
- 一次性生成多个回合或多个周期。
- 短板明显模板化或提前预设。
- 外部学习发生在短板之前。
- EVM 未真实调用却标记完成。
- GitHub 工厂未参与却标完整完成。
- 河图洛书本地文本链冒充真实多模型。
- 用户指出“没有感觉到执行”后仍继续推进。

## cron prompt 最小要求

cron prompt 必须写明：

- 本次只执行一个回合。
- 执行前读取 state.json。
- 若 state.next_round 不是本次回合，停止。
- 若上一回合未通过，停止。
- 执行完成后只写本回合证据。
- 不得生成未来回合或未来周期文件。
- 输出状态必须是：本回合通过 / 本回合失败 / 周期等待审计 / 周期降级停止。

## dry-run 测试规则

正式创建 cron 前，必须本地测试：

1. 初始化 state.json。
2. 模拟 R1 写入，检查 state 推进到 R2。
3. 尝试跳过 R2 写 R3，必须失败。
4. 尝试重复写 R1，必须失败。
5. 模拟 R2-R5 顺序写入，检查进入等待审计。
6. 模拟缺少 GitHub 工厂，周期审计应降级为“部分完成 / 证据不足”，但不标大错误。
7. 模拟一次性写多个回合，必须判定大错误。

## 最终边界

cron 能保证按时间触发和按状态推进，但不能单独证明智能真实进化。
真实完成仍以证据门禁为准。
